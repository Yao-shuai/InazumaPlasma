# ⚡ Inazuma Ecosystem (稻妻生态) 官方完全部署指南

欢迎来到 **Inazuma Ecosystem**！本项目是一个工业级的高性能 AI 网关与大模型代理计费系统。

### 🔥 核心架构特色（Hybrid 混合部署）

- **控制面 (Control Plane)：** 采用 Docker 容器化微服务架构（Nginx、AI Gateway、Chat 计费引擎、科技紫监控大屏），提供无缝的大模型路由与动态扩容能力。
- **数据面 (Data Plane)：** 底层缓存与持久化依托纯 C++ 打造的定制引擎 **InazumaKV**。进程直接裸跑在宿主机物理层，深度整合 Epoll 与 **AF_XDP 内核旁路技术**，实现主从零拷贝极速状态同步，彻底击穿传统 TCP/IP 协议栈的性能瓶颈！

------

## 🛠️ 0. 环境与底层依赖准备

由于本系统采用物理进程 + 容器微服务的混合架构，且涉及硬核的底层网络开发，请确保您的双机环境（Master/Slave）满足以下条件：

- **操作系统：** Ubuntu 22.04+ (强烈推荐)
- **内核版本：** Linux Kernel 5.15+ (必须支持 eBPF 和 AF_XDP)

### 0.1 容器环境准备

请确保系统已安装 Docker 和 Docker Compose 插件，用于承载上层微服务。

```
# 验证安装是否成功
docker -v
docker-compose version
```

### 0.2 C++ 底层与依赖库安装 (核心)

在**主从两台机器**上，必须安装完整的 C++ 编译链、eBPF 依赖，以及 InazumaKV 所需的高性能组件（如内存分配器 Jemalloc、向量检索库 Faiss、矩阵加速库 OpenBLAS 等）。

请依次执行以下命令，确保所有依赖一网打尽：

**第一步：安装基础编译链与网络库**

```
sudo apt update
sudo apt install -y build-essential cmake pkg-config git clang llvm
sudo apt install -y liburcu-dev libcurl4-openssl-dev ethtool libhiredis-dev
```

**第二步：安装高性能加速组件 (Jemalloc & OpenBLAS)**

```
sudo apt install -y libjemalloc-dev libopenblas-dev libomp-dev
```

**第三步：安装 Faiss 向量引擎 C++ 库**

```
sudo apt install -y libfaiss-dev
```

> **💡 提示：** 如果您的 Ubuntu `apt` 源中找不到 `libfaiss-dev`，请执行以下命令进行源码编译安装：
>
> Bash
>
> ```
> sudo apt install -y swig
> git clone https://github.com/facebookresearch/faiss.git
> cd faiss
> cmake -B build . -DFAISS_ENABLE_GPU=OFF -DFAISS_ENABLE_PYTHON=OFF -DBUILD_TESTING=OFF
> make -C build -j4
> sudo make -C build install
> cd .. && rm -rf faiss
> ```

**第四步：刷新系统动态库缓存** 安装完所有底层库后，务必刷新系统链接缓存，否则编译或运行时会报 `.so not found` 错误：

```
sudo ldconfig
```

### 0.3 核心 eBPF 依赖 (xdp-tools) 编译与安装

底层 AF_XDP 旁路加速的运行强依赖 `libxdp`。请在**主从两台机器**上依次执行以下命令进行编译安装：

Bash

```
cd ~
git clone --recurse-submodules https://github.com/xdp-project/xdp-tools.git
cd xdp-tools
./configure
make && sudo make install
sudo ldconfig  # 再次刷新系统动态库缓存
```

## 🔐 1. 全局环境与密钥配置 (.env)

为了保障大模型 API 资产与数据库的安全，本项目采用强环境隔离设计。**请不要在代码中硬编码任何敏感信息！**

1. 进入项目根目录。

2. 找到 `.env.example` 模板文件，复制并重命名为 `.env`：

   Bash

   ```
   cp .env.example .env
   ```

3. 使用文本编辑器（如 `vim .env`）打开 `.env` 文件，按说明填入您的专属配置：

   ```
   # ==========================================
   # ⚡ Inazuma Ecosystem 全局环境变量配置模板
   # 使用说明：请将此文件复制并重命名为 .env，然后填入真实的配置信息。
   # 注意：.env 文件包含敏感密码，绝不能提交到 Git 仓库！
   # ==========================================
   
   # ----------------------------------------------------
   # 1. 集群节点角色配置 (分布式部署必看)
   # ----------------------------------------------------
   # 决定当前机器运行 InazumaKV 的身份。
   # 填 master 则加载 master.conf (主节点，负责纯计算与网络克隆)
   # 填 slave  则加载 slave.conf  (从节点，负责灾备落盘与只读)
   # 默认为 master
   KV_ROLE=master
   
   # ----------------------------------------------------
   # 2. MySQL 数据库配置 (自动初始化使用)
   # ----------------------------------------------------
   # 系统首次启动时会自动使用这些信息为你建库建表，无需提前注册。
   DB_ROOT_PASSWORD=your_root_password_here
   DB_USER=inazuma_user
   DB_PASSWORD=your_db_password_here
   
   # ----------------------------------------------------
   # 3. JWT 鉴权安全校验密钥
   # ----------------------------------------------------
   # 不需要去任何网站申请！直接在键盘上随便敲一段极度复杂的乱码即可。
   # 这把“钥匙”将用于加密你的 Token，防止黑客伪造身份。
   JWT_SECRET=your_jwt_super_secret_string_here_!@#
   
   # ----------------------------------------------------
   # 4. 大模型 API Keys
   # ----------------------------------------------------
   # 必须去对应的云厂商官网申请真实的 API Key，根据你需要接入的模型至少填写一个。
   # QWEN_API_KEY=sk-xxxxxx
   # DEEPSEEK_API_KEY=sk-xxxxxx
   # OPENAI_API_KEY=sk-xxxxxx
   ```

## 🛡️ 2. Nginx 大屏安全锁配置 (.htpasswd)

本系统使用 Nginx 作为统一出口，并为内部的科技紫监控大屏 (Dashboard) 提供 Basic Auth 密码保护。

1. 进入 `inazuma-nginx` 目录。

2. 找到 `.htpasswd.example` 模板文件，复制并重命名为 `.htpasswd`：

   Bash

   ```
   cp inazuma-nginx/.htpasswd.example inazuma-nginx/.htpasswd
   ```

> **💡 提示**：默认的模板文件中包含了一对初始账号密码（账号：`admin`，密码：`inazuma2026`）。如果您想修改大屏密码，请在宿主机执行 `sudo apt install apache2-utils`，然后运行 `htpasswd -c inazuma-nginx/.htpasswd <您的新账号>` 进行重置。

## 🚀 3. 混合架构“两阶段点火” SOP

请严格遵循以下“先跑业务，后挂 XDP”的四步发车流程，不可乱序：

### ⚡ 第一阶段：拉起业务大盘与建立 TCP 握手

**1. [主机 Master] 拉起全矩阵：** 在主机 `inazuma-ecosystem` 根目录下，一键编译 C++ 底座并拉起 Docker 容器：

```
make clean
make
make run
```

*此时，物理层 KV 底座（绑核 CPU 0）和 Docker 业务群已就绪，等待从机连接。*

**2. [从机 Slave] 启动灾备底座：** 在从机 `inazuma-kv` 目录下，前台启动从库，完成 TCP 数据同步握手：

```
sudo taskset -c 0-3 ./bin/kvstore ./conf/slave.conf
```

*看见 `Reactor (Epoll) Server Started` 说明 1.4GB 共享内存（SHM）已开辟。*

### ⚡ 第二阶段：挂载 AF_XDP 核级旁路引擎 (无损热插拔)

**3. [从机 Slave] 挂载接收端 eBPF 钩子：** 在从机根目录打开新终端，执行：

```
ifname = eth_xdp
sudo ./enable_xdp_slave.sh
```

**4. [主机 Master] 挂载发送端 eBPF 钩子：** 在主机根目录打开新终端，执行：

```
ifname = eth_xdp
sudo ./enable_xdp_master.sh
```

*✅ 至此，底层网卡被彻底打通。您可以随时通过 `tail -f logs/xdp_gateway.log` 查看底层极速吞吐状态！*

## 🧹 5. 优雅熄火与系统清理

在 C++ 与 eBPF 混合架构中，安全退出极其重要，以防网卡死锁。

**[主机 Master] 一键安全熄火：** 在主机根目录执行：

```
make stop
```

*该指令会自动关停 Docker、向 C++ 进程发送析构信号，并强行清理网卡上残留的 eBPF 僵尸钩子，不留任何垃圾。*

*(从机 Slave：直接对运行中的 `kvstore` 进程使用 `Ctrl+C` 即可关闭。)*

## 🧪 6. 核心防线与场景压测矩阵 (Test Cases)

普通的 AI 缓存只能做简单的字符串匹配（傻瓜式命中），或者因为向量距离太近而引发“主宾反转”、“张冠李戴”的致命幻觉。 **Inazuma Gateway** 搭载了 C++ 内存级物理哨兵与 Python TF-IDF 知识库双重熔断防线。请打开终端查看网关日志，并使用以下 6 个极端场景对系统进行终极拷问：

### 场景一：高噪音抗干扰测试 (测试向量引擎鲁棒性)

- **Q1 (首次提问):** `用 C++ 写一个快速排序算法。`
- **Q2 (花式追问):** `兄弟，能麻烦你帮我用 C++ 代码手写一个快速排序算法吗，非常感谢了！`
- **🎯 预期表现：** 尽管 Q2 加了大量废话，但核心语义不变。网关将直接亮起绿灯 `📈 缓存命中 [Hits]`，计费打 1 折，客户端瞬间输出答案。

### 场景二：实体突变探测 (测试 Jieba 知识库熔断)

- **Q1 (首次提问):** `请告诉我 Redis 的默认运行端口是多少？`
- **Q2 (诱导突变):** `请告诉我 MySQL 的默认运行端口是多少？`
- **🎯 预期表现：** 两个句子的向量距离 (L2 Distance) 极近，但后端的 `jieba` 引擎会敏锐抓取到核心名词变了（Redis 变成了 MySQL）。终端将报出 `🚨 [知识库熔断] 实体不匹配`，系统拒绝给出错误缓存，强制穿透回源。

### 场景三：极速否定意图拦截 (测试 C++ 底层物理哨兵)

- **Q1 (首次提问):** `如何在 Linux 内核中开启 io_uring？`
- **Q2 (强否定词):** `如何在 Linux 内核中禁止 io_uring？`
- **🎯 预期表现：** 根本不需要走到 Python 深度学习那一层，C++ 底层哨兵在内存扫描时瞬间捕捉到“禁止”二字产生状态冲突，直接触发物理熔断，防止 AI 教用户错误的操作。

### 场景四：逻辑/方向反转防御 (测试动态阈值门控)

- **Q1 (首次提问):** `从上海飞往北京的航班一般需要几个小时？`
- **Q2 (主宾反转):** `从北京飞往上海的航班一般需要几个小时？`
- **🎯 预期表现：** 词汇 100% 重合，但方向反了！C++ 哨兵检测到高度重合却不完全一致，将拦截门槛临时暴升至 `0.99`。Reranker (精排) 模型给出打分后被高门槛无情拦截，强制回源以确保逻辑严谨。

### 场景五：多租户模型沙盒隔离 (测试防止“串台”)

- **操作：** 在 Chatbox 中将模型切换为 `qwen-turbo`。
- **Q1:** `你是什么模型？` *(Qwen 会回答我是通义千问)*
- **操作：** 将模型切换为 `deepseek-chat`，问完全相同的问题。
- **Q2:** `你是什么模型？`
- **🎯 预期表现：** 终端将打印 `🛡️ [模型沙盒隔离] 发现异构缓存，拒绝串台！`。即使语义 100% 一致，DeepSeek 也绝不会拿到 Qwen 的缓存答案，彻底杜绝企业级多租户数据污染。
